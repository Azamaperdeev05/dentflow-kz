# 07. Бизнес-логика

## 1. Тіркелу және верификация логикасы

1. Пайдаланушы форма толтырады.
2. Валидациядан өткен соң, құпиясөз hash жасалады.
3. Растаушы код генерацияланады және hash түрде сақталады.
4. `PendingRegistration` кестесіне уақытша дерек жазылады.
5. Код email арқылы жіберіледі.
6. Пайдаланушы кодты енгізгенде тексеріліп, негізгі `User` және профиль (`PatientProfile`/`DoctorProfile`) құрылады.

Ескерту:

- Дәрігер үшін `specializations` JSON форматта сақталады.

## 2. Кіру және 2FA

- Credentials логині email+password арқылы жүреді.
- Парольді тексеру `bcrypt` арқылы іске асады.
- 2FA қосулы болса, TOTP код валидациясы міндетті.
- Сәтті/сәтсіз логин әрекеттері қауіпсіздік логына жазылады.

## 3. Қабылдау (Appointment) логикасы

### Пациент сценарийі

- Дәрігер таңдайды.
- Күн/уақыт ұяшығын таңдайды.
- Жүйе уақыт қолжетімділігін тексереді.
- Appointment жазбасы жасалады.

### Дәрігер сценарийі

- Қабылдауды растайды/болдырмайды/аяқтайды.
- Өзгерістер пациентке хабарландыру тудырады.

## 4. Емдеу және төлем логикасы

- Дәрігер диагноз, процедура сатылары және құнын береді.
- Жүйе толық құнды есептейді (`sumStageCosts`).
- Пациент емдеу жоспарын approve/reject жасай алады.
- Төлем енгізілгенде `paidAmount` жаңарады.
- Қалдық төлем: `totalCost - paidAmount`.

## 5. Чат және хабарландыру логикасы

- Чат тек рұқсат етілген user жұбына ашылады (`canUsersChat`).
- Хабар жіберу/оқу кезінде `isRead` статусы қолданылады.
- Бизнес оқиғалар негізінде notification құрылады.

## 6. Қауіпсіздікке қатысты бизнес ережелер

- Рөлдік шектеу middleware + server-side require функцияларымен тексеріледі.
- Әр маңызды өзгеріс аудит логқа түседі.
- Күдікті логин әрекеттері risk signal-ға жазылады.

## 7. Логикалық тұтастық

- Domain ережелері API ішінде орталықтандырылған.
- Валидация қайталанбайтын және бірізді (Zod схемалары).
- Қате жағдайлар анық кодтармен қайтарылады (400/403/429/500).
