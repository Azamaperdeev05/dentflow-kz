# 03. Талаптар және Use Case моделi

## 1. Функционалдық талаптар

### 1.1 Аутентификация

- Пайдаланушы тіркелуі тиіс (пациент/дәрігер).
- Email растау коды арқылы верификация жасалуы тиіс.
- Құпиясөзді ұмыту және қалпына келтіру ағындары болуы тиіс.
- 2FA (TOTP) қосу/өшіру мүмкіндігі болуы тиіс.

### 1.2 Пациент модулі

- Дәрігерлерді іздеу және фильтрлеу.
- Қабылдауға жазылу, ауыстыру, бас тарту.
- Чат арқылы дәрігермен байланыс.
- Медициналық файл жүктеу.
- Емдеу жоспарын қарау және бекіту/қабылдамау.

### 1.3 Дәрігер модулі

- Қабылдауларды көру және статус өзгерту.
- Емдеу жоспарын құру.
- Төлем енгізу және қаржы көрсеткіштерін жаңарту.
- Пациенттер тізімін сүзгілеу.

### 1.4 Админ модулі

- Қауіпсіздік журналын көру.
- Күдікті әрекеттерді бақылау.
- Әрекеттерді фильтрлеу (күні, action, статус, user, suspicious).

## 2. Функционалдық емес талаптар

- Жүйе жауап беруі тұрақты және қате өңдеуі түсінікті болуы керек.
- Валидация барлық енгізу нүктелерінде орындалуы керек.
- Рөлдік рұқсат бақылауы міндетті.
- Қауіпсіздік оқиғалары журналға жазылуы тиіс.

## 3. Use Case тізімі

1. Пациент тіркеледі және аккаунтты растайды.
2. Дәрігер тіркеледі және мамандықтарды таңдайды.
3. Пациент дәрігерге жазылады.
4. Дәрігер қабылдауды растайды/аяқтайды/болдырмайды.
5. Дәрігер емдеу жоспарын жасайды.
6. Пациент емді бекітеді.
7. Дәрігер төлем енгізеді.
8. Пациент пен дәрігер чат жүргізеді.
9. Админ аудит журналын қарайды.

## 4. Актерлер және жауапкершілік

- Пациент: өз деректері мен жазылуына жауапты.
- Дәрігер: клиникалық және қаржылық жазбалардың дұрыстығына жауапты.
- Админ: қауіпсіздік бақылауына жауапты.

## 5. Қабылдау критерийлері

- Барлық негізгі API endpoint-тер қолжетімді және қате жағдайын өңдейді.
- Build қатесіз өтуі керек.
- DB schema талаптармен сәйкес болуы керек.
- Тіркелу/кіру/жазылу/емдеу процестері логикалық түрде аяқталуы керек.
